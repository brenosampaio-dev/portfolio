## BLOCO 1 — Context & Problem

Field service technicians — the people who install HVAC systems, maintain industrial equipment, repair utilities infrastructure — spend a disproportionate share of their working day on documentation.

Work orders, service reports, material logs, sign-offs. In most operations, these are either paper forms completed at the end of a shift or desktop-first software ported to mobile with minimal redesign. Both fail in similar ways: fields are skipped, forms are incomplete, sign-offs are delayed, and back-office teams spend significant time correcting or chasing field documentation before it can be processed.

Research on field service management consistently surfaces the same four root causes:

**Form abandonment under task pressure.** Technicians complete documentation between physical tasks, standing at a job site, often one-handed. A 35-field work order with no guidance on what's required produces skipped fields and incomplete submissions — not because technicians are careless, but because the interface gives them no structure to follow under real conditions.

**Silent connectivity failures.** Field environments — basements, concrete industrial facilities, remote sites — routinely have no connectivity. Tools that assume a live connection either fail silently (data lost), show an error with no actionable path, or block the technician from completing their work. The technician can't do anything about the signal. The interface should be designed around that reality.

**Sign-off friction.** Customer or site supervisor signature is a standard work order requirement. In most systems, getting a signature means navigating to a specific screen, handing the device to a customer who didn't expect it, and returning to a workflow the technician has partially lost context on. The handoff is awkward; the context loss is real.

**Data quality at source.** Back-office error correction is expensive. The root cause is an intake form that collects errors rather than preventing them — no field-level validation, no required-field enforcement, no guidance at entry.

FieldOps was designed to address all four.

> *Pull-out:* The interface should be designed around the conditions where the work actually happens — not around the conditions where it's easiest to design software.

## BLOCO 3 — Process & Key Decisions

### Workflow mapping before wireframing

Before any screen design, I mapped the technician's workflow as a sequence of jobs-to-be-done: not features to support, but moments where the tool either creates clarity or gets in the way.

The critical path: receive order → review requirements → travel to site → execute work → document → get signature → close out → sync.

Each step has a different device context (driving vs. on site), a different information need (what am I going to vs. what did I do), and a different failure mode (missed connectivity during sync, skipped required field, signature screen timing out while the customer holds the phone).

Mapping this before wireframing prevented the failure mode most common in B2B mobile: a tool that covers every feature but serves no single moment particularly well.

---

### Key Decision 1 — Guided, not linear

Early wireframes treated the work order as a scrollable long-form — all fields visible, technician scrolls and fills. This was wrong for an obvious reason once articulated: presenting 35 fields as a scroll gives the technician no information about what's required, no sense of progress, and no structure to follow under task pressure. Fields get skipped because the interface offers no guidance on which ones matter.

The final design uses a **GuidedStep model**: one task per screen, with a clear label, explicit completion requirements, and a step counter showing current position and total. Required fields are surfaced before optional ones. Conditional fields appear only when a prior answer triggers them — the form is adaptive, but the complexity is invisible to the technician.

The guided model also changes the relationship between completion enforcement and friction. In a scroll form, blocking advancement on a required field feels adversarial. In a guided form, the step structure makes the requirement feel like natural sequence. The same enforcement, different experience.

---

### Key Decision 2 — Offline must be silent by default

The first offline indicator design was a banner at the top of the screen reading "No Connection." It was removed immediately.

A "No Connection" banner creates anxiety and implies that the user should stop — that continuing might cause data loss. But the entire point of offline-first architecture is that the user doesn't need to stop. The UI was actively undermining the product's core reliability guarantee.

The final design uses a persistent but minimal **OfflineIndicator** in the navigation bar — an icon with a short label that cycles through three states: `offline` ("Saving locally"), `syncing` ("Uploading..."), `reconnecting` ("Reconnecting..."). When offline, the label communicates what is happening (saving) rather than what is absent (internet). No banner, no modal, no block.

The system escalates offline visibility at exactly one moment: when a workflow step requires real-time validation that can't run locally. At that point, a contextual inline notice explains the specific step and offers to queue it for when connection restores. This is the only moment offline state takes up real UI space — because it's the only moment it actually requires a decision from the user.

---

### Key Decision 3 — Signature is a workflow step, not a screen

The initial signature design was a dedicated screen reached via a "Get Signature" button at the bottom of the work order. The problem: this turns signature collection into an interrupt. The technician hands the device to a customer who didn't expect it, mid-workflow. The customer signs something they didn't navigate to. The technician returns to a workflow position they've lost context on.

In the final design, signature collection is a step inside the GuidedStep system — it appears in sequence after the last documentation step, at the natural point where the technician has finished their work and is ready for close-out. The handoff happens at a moment that makes sense to both parties. The SignatureCanvas component takes the full usable screen area. After signature, the device returns to the technician at the confirmation step, with the next action immediately visible.

One additional detail: the re-sign affordance is a clearly labeled button below the signature area — not a small "X" on the signature itself. Customers in a hurry sign badly. The technician needs to be able to ask for a re-sign without explaining how to clear the canvas.

---

### Key Decision 4 — The job queue is an intent surface, not a data view

Early designs showed all available order metadata at the list level: order ID, address, customer name, equipment type, scheduled time, status, assigned materials.

Information overload at the list level creates scanning effort at exactly the moment when the technician's working memory is already loaded with physical task context. They don't need all the data at the queue level — they need to answer two questions quickly: "What's next?" and "Is anything blocked or urgent?"

The final **OrderCard** design surfaces three things at the list level: status (color-coded, immediately scannable), customer name and address (where am I going), and scheduled time (when). Everything else lives in the order detail. Next-order selection is a 2-second scan, not a 20-second read.

## BLOCO 6 — Outcome & Reflection

### What this case demonstrates

Every design decision has a "why" traceable back to the workflow map, the constraint set, or a specific failure mode being prevented:
- **Guided over linear:** task pressure and skipped fields
- **Silent offline:** the banner was undermining the product's reliability guarantee
- **Signature as a step:** the interrupt model created friction for both parties
- **Queue surfaces three things:** scanning speed matters more than information completeness at list level

### Honest hypotheses (not results)

- Guided steps would reduce form abandonment vs. scroll forms — testable with completion rate comparison
- Silent offline model would reduce anxiety-driven workflow interruptions — observable in usability testing
- Signature as a workflow step would reduce signature-related support tickets
- These predictions are reasonable. They are not results.

### What I'd do differently

The guided step model optimizes for first-time and infrequent users. It creates friction for experienced technicians who've completed the same order type dozens of times. A future iteration would add a **review mode** — all fields visible, directly editable — for technicians above a completion threshold for a given order type.

The "Saving locally" label assumes trust in local storage. On older Android hardware where apps close unexpectedly, some technicians won't trust it. Adding "Last saved X minutes ago" to the offline indicator would address this without adding visual noise.

### Workflow com AI

Claude foi usado para:
- Pesquisa de domínio: failure modes em field service management, padrões de workflow de técnicos, o que as ferramentas de FSM erram consistentemente
- Geração de edge cases: o que acontece se um step obrigatório é pulado, se o device reinicia no meio do form, se a assinatura é capturada offline e o sync falha
- Rascunho de error states, empty states, offline notices e mensagens de confirmação
- Crítica da lógica de workflow antes do design começar — identificou o problema da "signature como interrupt" antes de qualquer tela ser feita

Nenhuma tela foi gerada por AI. Todas as decisões de workflow e design são do designer.
